## Known Pitfalls

- **[HIGH]** Telemetry counters (search_queries_total, servers/agents/skills counts) are lifetime cumulative snapshots embedded in event payloads (startup events), not incremental deltas from heartbeat events. Summing these values across multiple heartbeat or event rows massively over-counts actual activity. Heartbeat opt-in is 0%, so search query data originates from startup event payloads, not heartbeats.
  - Resolution: When aggregating telemetry metrics, treat snapshot counters as point-in-time values rather than summable deltas. For lifetime counters, take the MAX or LAST value per entity per time window instead of SUM. Verify the source event type (startup vs. heartbeat) before choosing an aggregation strategy, and document which fields are snapshots vs. true deltas.
- **[HIGH]** When aggregating cumulative counters (e.g., totals that accumulate over time or across instances), using sum() instead of max() produces inflated/incorrect results. Cumulative counters should be aggregated with max() per instance to get the final value, then summed across instances if needed.
  - Resolution: Identify whether each metric is a cumulative counter or a rate/gauge. For cumulative counters, use max() to extract the final value per instance before any cross-instance aggregation. Rewrite aggregation functions like _compute_search_stats to handle this distinction explicitly. After fixing, re-run analysis scripts and verify output tables to confirm correctness.
- **[HIGH]** Inserting a new slide into the presentation deck causes duplicate data-slide attribute indices and mismatched slide number display labels in both HTML files, as all subsequent slides retain their original numbering after insertion.
  - Resolution: After inserting any new slide, run a Python script to sequentially renumber all data-slide attributes and visible slide number labels across both files, covering the full 17-slide deck to ensure consistency.
- **[HIGH]** Integration tests for SyncOrchestrator still reference the old API signature that accepted credential_helper and token_manager parameters. After SyncOrchestrator was refactored to a manifest-based flow, the integration test file was not updated in sync, causing 17 test failures due to API mismatch.
  - Resolution: When refactoring SyncOrchestrator's API (removing credential/token params, adding manifest-based flow), update the integration test file in the same change to match the new constructor/call signatures. Search for all usages of credential_helper and token_manager params in test files and replace with the manifest-based equivalents.
- **[HIGH]** Dashboard.tsx defines its own local Agent interface (line 55) that does not include all fields present in the server stats data (e.g., ans_metadata). When agentsFromStats maps over server data, any field not declared in Dashboard.tsx's local interface is silently dropped, causing downstream components like AgentCard to never receive those fields even though they exist in the raw data. Additionally, Dashboard.tsx uses project-specific prop names ('agentApiToken', 'refreshData()') rather than what might be assumed from other conventions ('authToken', 'refreshStats()').
  - Resolution: When adding new fields to the server stats data model, always update Dashboard.tsx's local Agent interface AND the agentsFromStats transform function simultaneously. When in doubt about prop names used in Dashboard.tsx, read the file directly rather than assuming naming conventions from other parts of the codebase.
- **[HIGH]** When using AsyncMock in Python's unittest.mock, leaving a nested method (e.g., .head) unmocked can cause infinite recursion when code accesses attributes like .status_code on the returned AsyncMock. AsyncMock auto-creates child mocks for attribute access, and certain fallback paths (like a HEAD request fallback) can recurse infinitely through these auto-generated mock attributes.
  - Resolution: Explicitly set side_effect (e.g., side_effect=SomeException) or return_value on all mock methods that can be reached by fallback or retry paths in the code under test. Do not rely on AsyncMock's default auto-speccing behavior for methods that trigger branching logic based on response attributes like .status_code.
- **[HIGH]** Assuming a specific PR (e.g., PR #737) fully resolved a reported visibility inconsistency bug. The fix was incomplete — the issue was narrowly scoped in the PR but actually affects 50+ locations across 28 files project-wide, leading to a false sense of resolution.
  - Resolution: Before closing a bug report as fixed, perform a broad codebase search (e.g., grep/AST search) for all instances of the problematic pattern — not just the files mentioned in the PR. Verify coverage is complete, not just the reported examples.
- **[HIGH]** Git tags used as stable references for documentation (e.g., fix guides with exact line numbers) become stale when new commits are merged after the tag is created. The tag must be moved to the new HEAD, but this creates a moving-target problem: any documentation or guides that hardcode line numbers against a specific tag will break if the tag is later moved to a different commit where line numbers have changed.
  - Resolution: Establish a clear freeze point: once documentation referencing exact line numbers is committed and the tag is finalized, do not move the tag again. If new commits must be included, create a new patch tag (e.g., v1.0.19) rather than moving the existing one. Alternatively, use commit SHAs instead of mutable tags for line-number-dependent references, and only use tags for human-readable release identification.
- **[HIGH]** Calling the semantic search API fails in multiple ways: nginx returns 400 errors when the token format is wrong (the token file contains JSON, not a bare token string), and authentication against the ECS registry endpoint with a Bearer token yields 500/401 from nginx. Additionally, attempting to bypass nginx by connecting directly to the ALB fails SSL certificate verification.
  - Resolution: Parse the JSON token file to extract the raw token string before using it as a Bearer token. Do not attempt direct ALB connections as SSL verification will fail. Ensure requests are routed through the correct nginx-fronted endpoint with a properly formatted Authorization header derived from the parsed JSON token file.
- **[HIGH]** Puppeteer-based PDF generation fails in two common ways: (1) the Puppeteer module cannot be found via require due to installation or path issues, and (2) even when the module loads, Puppeteer cannot locate a compatible Chrome/Chromium binary at the expected version path.
  - Resolution: Ensure Puppeteer is properly installed (npm install puppeteer, not just puppeteer-core) so it downloads its bundled Chromium. If using puppeteer-core or a custom environment, explicitly set the executablePath option to the correct Chrome binary. In containerized or CI environments, pre-install Chromium and pass its path via PUPPETEER_EXECUTABLE_PATH env var or the launch options. Verify version compatibility between the Puppeteer package and the installed Chrome binary.
- **[HIGH]** Keycloak persists its realm configuration (including frontend URL, OAuth client redirect URIs, and issuer URLs) in its database at initialization time. When the deployment URL changes (e.g., switching from a CloudFront distribution to a custom domain, or rotating CloudFront distributions), simply updating environment variables is insufficient — the database retains stale URLs, causing OAuth login failures due to redirect URI mismatches and issuer validation errors. Additionally, using axios.post() for logout triggers CORS errors because browsers block cross-origin redirect following on XHR/fetch requests, which Keycloak's logout endpoint relies on.
  - Resolution: 1) When changing the Keycloak frontend URL or OAuth client redirect URIs, either: (a) destroy and recreate the RDS database + ECS service so Keycloak reinitializes with the correct URLs, OR (b) use the Keycloak Admin API or Admin Console to manually update all realm frontend URLs and client redirect URIs before switching traffic. 2) Always audit ALL Keycloak client configurations for stale domain references, not just environment variables. 3) For logout, use a browser navigation redirect (window.location.href) instead of axios.post() to avoid CORS errors when Keycloak issues cross-origin redirects during the logout flow.
- **[HIGH]** Configuration changes to Keycloak client secrets or other auth credentials are not picked up by running services until they are explicitly restarted. Services (auth, registry) cache secrets at startup, so updates to AWS Secrets Manager or Keycloak do not take effect in live containers without a restart/rebuild.
  - Resolution: After updating secrets in AWS Secrets Manager or regenerating Keycloak client secrets, force-restart all dependent services (auth service, registry service) to trigger a container rebuild and reload the new credentials. Verify login/OAuth2 flows work post-restart.
- **[HIGH]** When a Docker container uses a baked-in image (rather than volume-mounted source), restarting the container does not apply local code changes. Additionally, Docker layer caching can cause builds to use stale code if cache invalidation is not forced, resulting in a running container whose files differ from the local source.
  - Resolution: Rebuild the Docker image from scratch (e.g., `docker build --no-cache`) after making code changes, then redeploy the container. Alternatively, use volume mounts during development so local changes are reflected immediately without rebuilding. Verify deployed code matches source by comparing file checksums between the container and local files.
- **[HIGH]** Code fixes applied to source files are not reflected in running Docker containers unless the container is rebuilt. Simply restarting the container or syncing data will not pick up code changes if the application code is not volume-mounted into the container — the old image layers with the original code continue to run.
  - Resolution: After modifying application code, rebuild the Docker image (e.g., `docker compose build` or `docker build`) before restarting the container. If the fix involves data migration, also re-sync or re-seed the relevant data (e.g., delete and re-create affected records) after the rebuilt container is running.
- **[HIGH]** When switching authentication providers (e.g., from Keycloak to Entra ID or vice versa), persisted service registry cards in MongoDB retain the OAuth2/OIDC endpoint configuration from the previously active provider. The registry card is written at initialization time and is not automatically updated when the AUTH_PROVIDER environment variable or equivalent config changes, causing stale auth config to be served.
  - Resolution: After changing the AUTH_PROVIDER configuration, delete the stale registry card(s) from MongoDB manually (e.g., via a cleanup script) and restart the registry service to force re-initialization with the new provider's endpoints. Alternatively, implement a startup check that validates the persisted auth provider config against the current AUTH_PROVIDER value and triggers re-initialization on mismatch.
- **[HIGH]** The ANS badge disappears or never populates because of multiple gaps in the data pipeline: (1) the ANS API client calls GET /v1/agents/{ans_agent_id} with a raw ans:// URI instead of a UUID, causing 404s; (2) ans_metadata is not propagated through intermediate data transformation steps; (3) the ans_sync_scheduler or ans_service fails silently, leaving MongoDB documents without ANS metadata even after successful ANS linking.
  - Resolution: Add a _resolve_ans_id helper that converts ans:// URIs to UUIDs by paginating the ANS API's agent list endpoint before making detail calls. Audit every stage of the data pipeline (API response → transformation → MongoDB write → registry API response) to ensure ans_metadata is explicitly passed through. Add logging at each stage and query MongoDB directly to verify ans_metadata is persisted, not just resolved in memory.
- **[HIGH]** ANS integration environment variables (ANS_INTEGRATION_ENABLED and related vars) must be explicitly declared in BOTH docker-compose.yml for local development AND the ECS task definition for deployed environments.

The Problem

Coding agents have amnesia.

Every Claude Code session starts from scratch. The agent reads the same files, hits the same dead ends, and rediscovers the same architectural quirks — over and over.


The reasoning that produced last week's fix is locked inside a JSONL transcript that no one reads.

  • No memory of past dead-ends or pivots
  • No accumulated knowledge of project quirks
  • No diagnostic patterns from prior debugging
  • Every agent is a new hire on day one
02 / 11
The Approach

Treat each session as a reasoning trace.

Extract a reasoning DAG from every session — nodes for hypotheses, dead ends, pivots, discoveries, and solutions. Cluster across sessions to distill durable knowledge: insights, pitfalls, diagnostic strategies.

04 / 11
Two-Tier Extraction

Always-on speed, on-demand quality.

Tier 1 · Real-time

Warm Extractor

Regex heuristics: keyword scoring, error-resolution pairing, explicit conclusion detection. Three strategies merged with corroboration boost.

  • < 1 second per session
  • Zero API cost
  • Confidence: 0.25 – 0.75
  • Runs on every session-end
Tier 2 · Batch consolidation

Cold Extractor

Claude Sonnet 4.6 classifies overlapping 5-message windows. A second pass infers semantic edges between nodes.

  • Concurrent LLM calls
  • Confidence: 0.5 – 0.95
  • Replaces warm nodes via dedup
  • Runs nightly or on demand
05 / 11
Knowledge Distillation

From nodes to durable knowledge.

Agglomerative clustering on sentence embeddings groups semantically similar reasoning across sessions. Each cluster is classified by an LLM into one of three knowledge types.


Semantic deduplication (cosine ≥ 0.85) prevents memory pollution as the store grows.

06 / 11
Auto-Discovery

Projects describe themselves.

A .cognitive/ folder in the repo root makes the project self-describing. Stable project IDs from repo_name + sha256(README)[:6] work across machines without hardcoding.


Session-start hooks load the cached profile in <500ms — no ChromaDB query, no API call.

# .cognitive/ ├── manifest.json # project identity ├── config.json # auto_ingest, thresholds ├── cached_profile.md # fast-load snapshot └── llms.txt # agent-readable metadata $ cmm init . Initialized: supply-chain-a3f2b1 $ cmm hook start → context injected (12 nodes, 3 sessions)

08 / 11
Evaluation

Three signals for helpfulness.

Signal A

Errors Resolved

DEAD_END → memory retrieval → SOLUTION within 8 messages. Did memory help fix it?

Signal B

Pitfalls Avoided

Surfaced pitfall has no matching DEAD_END node in the session. Was the trap dodged?

Signal C

Pivots After Retrieval

PIVOT node within 5 messages of a search. Did memory change direction?

Sessions split by memory_used_at_start — compare dead ends, pivots, messages-to-solution between cohorts.
